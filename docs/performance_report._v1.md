# MileDay 기능·성능 테스트 분석 보고서

> 작성일: 2026-07-10  
> 분석 대상: Playwright 기반 Electron 개발 모드 연속 반복 테스트  
> 목적: 현재 구현의 성능 기준선 수립, `success=false` 원인 분석, 개선 전·후 정량 비교 기반 마련

## 1. 분석 요약

| 핵심 지표 | 결과 |
|---|---:|
| 테스트 시나리오 | 5개 |
| 시나리오별 반복 | 25회 |
| 전체 실행 | 125회 |
| 성공 | 114회 |
| 실패 | 11회 |
| 전체 성공률 | 91.2% |
| 기록된 API 호출 | 823회 |
| HTTP 200 | 787회 |
| HTTP 502 | 36회 |
| API 호출 기준 502 비율 | 4.37% |
| 502가 포함된 실패 실행 | 8회 |
| 502가 없는 타임아웃 실패 | 3회 |

현재 구현은 **쓰기 요청 자체는 안정적**이었다. 기록된 `POST /goals`, `POST /goals/{goal_id}/milestones`, `PATCH /milestones/{id}/complete`, `PATCH /settings`는 모두 HTTP 200을 반환했다. 반면 쓰기 이후 화면을 갱신하기 위해 호출되는 조회 API에서 502가 발생했고, 일부 실행에서는 API가 정상 처리되었음에도 Playwright가 15초 안에 목표 UI 요소를 찾지 못했다.

성능 측면에서는 `/calendar/month`와 `/calendar/date/{date}`가 가장 큰 병목이다. 두 API는 정상 응답만 집계해도 중앙값이 약 4.1초이며, P95는 10초를 넘는다. 따라서 현재 지연은 단순히 테스트 타임아웃이 짧아서 발생한 문제가 아니라, **캘린더 조회 처리와 중복 재조회 구조를 우선 개선해야 하는 상태**로 판단된다.

## 2. 테스트 환경

| 항목 | 내용 |
|---|---|
| 애플리케이션 | MileDay |
| 클라이언트 실행 방식 | Electron 개발 모드 |
| 프론트엔드 테스트 | Playwright |
| 백엔드 | 로컬 PC의 FastAPI |
| 데이터베이스 | Supabase PostgreSQL |
| 프로세서 | AMD Ryzen 5 7640HS with Radeon 760M Graphics, 4.30 GHz |
| 메모리 | 16.0 GB, 사용 가능 15.3 GB |
| 실행 방식 | 시나리오별 25회 연속 실행 |
| 반복 간 대기 | 없음 |
| 서버 프로세스 | 모든 테스트에서 동일 프로세스 유지 |
| 실패 제한 시간 | UI 요소가 15,000ms 안에 표시되지 않으면 실패 |
| 테스트 목적 | 사용자 체감 응답시간 단축 및 포트폴리오용 개선 전·후 비교 |

### 2.1 통계 해석 시 주의사항

- 로컬 Electron과 FastAPI를 사용하지만 DB는 외부 Supabase에 있으므로 네트워크 왕복시간과 외부 서비스 상태가 포함된다.
- 반복 사이 휴지 시간이 없어 실제 일반 사용보다 가혹하지만, 연속 요청과 연결 재사용 안정성을 검증하는 기준선으로 활용할 수 있다.
- 실패 실행에는 `duration_ms`가 없으므로 시나리오 응답시간 통계는 **성공 실행만 대상**이다. 실제 사용자 체감은 표의 수치보다 나쁠 수 있다.
- 502는 약 10~30ms에 빠르게 반환된 경우가 많아 API 평균을 왜곡한다. API 응답시간 통계는 HTTP 200만 대상으로 계산했다.

## 7. 502 원인 분석

### 7.1 직접 원인

FastAPI 로그의 최종 예외는 다음과 같다.

```text
httpcore.RemoteProtocolError: Server disconnected
```

Traceback은 `httpx → httpcore → HTTP/2 connection pool`에서 응답을 수신하는 도중 원격 서버가 연결을 종료했음을 보여준다. 따라서 로컬 FastAPI가 요청을 받지 못한 것이 아니라, **FastAPI가 Supabase 계층에 HTTP 요청을 보내고 응답을 기다리는 과정에서 원격 연결이 종료된 것**으로 해석할 수 있다.

```text
Electron/Playwright
→ 로컬 FastAPI
→ FastAPI의 Supabase HTTP 요청
→ HTTP/2 응답 수신 중 원격 연결 종료
→ RemoteProtocolError
→ FastAPI가 502 반환
→ UI 갱신 데이터 누락
→ Playwright 15초 대기 후 실패
```

### 7.2 원인 수준별 판단

| 수준 | 원인 | 판단 강도 | 근거 |
|---|---|---|---|
| 직접 원인 | HTTP/2 응답 수신 중 원격 연결 종료 | 확정 | `RemoteProtocolError: Server disconnected` |
| 결과 | 처리되지 않은 외부 통신 예외가 502로 변환 | 확정 | `unhandled server error`와 접근 로그 |
| 유발 조건 | 대기 없이 연속·병렬 조회가 동일 풀에 집중 | 가능성 높음 | 25회 연속 실행, 한 동작당 평균 6.58회 호출 |
| 구조적 원인 | 변경마다 다수의 전체 조회 실행 | 가능성 높음 | 여러 GET이 한 묶음으로 반복 |
| 외부 요인 | Supabase 또는 중간 네트워크가 연결 종료 | 가능성 있음 | 원격 종료는 확인되나 종료 주체는 미확정 |
| PC 자원 부족 | 로컬 CPU·RAM 부족 | 가능성 낮음 | 메모리 오류가 아니라 HTTP 프로토콜 계층 예외 |
| DB 풀 고갈 | PostgreSQL 직접 연결 수 초과 | 근거 부족 | SQL 드라이버가 아닌 `httpx/httpcore`에서 발생 |

### 7.3 502가 빠르게 반환된 이유

실패 GET 다수는 약 10~30ms 안에 502를 반환했다. 장시간 쿼리 후 타임아웃이라기보다 **재사용하려던 HTTP/2 연결이 이미 종료되었거나 응답 스트림이 즉시 끊어진 패턴**에 가깝다.

여러 조회가 동시에 502가 된 것은 각 라우터 로직이 동시에 잘못되었다기보다 공통 Supabase 클라이언트 또는 연결 풀의 장애가 여러 엔드포인트에 전파된 것으로 보는 편이 타당하다.

## 8. 15초 타임아웃 원인 분석

### 8.1 502가 포함된 경우

POST/PATCH가 성공해도 UI 갱신용 GET이 502이면 로컬 상태가 최신 데이터를 받지 못한다. Playwright는 새 제목이나 변경된 상태를 기다리지만 데이터가 들어오지 않아 15초 후 실패한다. 이 경우 15초는 원인이 아니라 **앞단 장애가 드러나는 최종 시점**이다.

### 8.2 502가 없는 경우

| 관찰 | 가능한 원인 |
|---|---|
| 목표 POST와 일부 GET은 200이나 제목 미표시 | 재조회 누락, 목록 필터·정렬, 렌더링 경쟁, locator 불일치 |
| 날짜 API와 월 API가 200이나 날짜 미표시 | 이전 요청의 늦은 응답이 최신 선택을 덮는 경쟁 상태 |
| 예상 후속 GET이 일부만 기록 | 이벤트 흐름 중단 또는 컴포넌트 상태 전환 누락 |
| `exact: true` 텍스트 탐색 실패 | 공백·접두어·중첩 DOM·날짜 표시 형식 차이 |

날짜 선택 기록에는 이전 날짜 요청과 현재 날짜 요청이 함께 존재한다. 이전 요청이 나중에 완료되면 최신 상태를 과거 응답이 덮어쓸 수 있으므로 요청 취소나 최신 요청 식별이 필요하다.

## 9. 개선 방안

### 9.1 우선순위

| 우선순위 | 개선 항목 | 기대 효과 |
|---:|---|---|
| P0 | `RemoteProtocolError` 전용 제한적 재시도 | 순간 연결 종료로 인한 502 감소 |
| P0 | 월·일 캘린더 쿼리 최적화 | 전체 E2E 지연과 15초 초과 감소 |
| P0 | 전체 재조회 대신 로컬 상태 즉시 갱신 | 토글·생성 체감시간 대폭 단축 |
| P1 | 중복 API 호출 제거·요청 병합 | 외부 연결 부하와 경쟁 감소 |
| P1 | 이전 날짜 요청 취소·최신 응답만 반영 | 날짜 선택 경쟁 상태 제거 |
| P1 | 502와 UI 타임아웃을 별도 코드로 기록 | 개선 효과를 원인별로 측정 |
| P2 | `data-testid` 기반 locator | 테스트 오탐 감소 |

| P2 | request ID와 외부 호출 구조화 로그 | 장애 추적 시간 단축 |

### 9.2 502 대응

#### 제한적 재시도

| 항목 | 권장 |
|---|---|
| 최대 시도 | 최초 1회 + 재시도 1~2회 |
| 대기 | 100ms → 300ms |
| 우선 적용 | 멱등 GET |
| POST/PATCH | 중복 쓰기 위험 때문에 자동 재시도 지양 |
| 연결 처리 | 프로토콜 오류 시 기존 연결 폐기 후 새 연결 사용 |
| 로깅 | 요청, 시도 횟수, 예외, 총 소요시간 기록 |

재시도는 근본적인 중복 요청 문제를 숨기지 않도록 제한적으로 사용해야 한다.

#### HTTP 클라이언트 수명주기

- 요청마다 Supabase 클라이언트를 생성한다면 앱 수명주기 단위 재사용을 검토한다.
- 끊어진 연결을 계속 재사용한다면 예외 발생 시 클라이언트 또는 풀을 폐기하고 재생성한다.
- HTTP/2 문제 분리를 위해 개발 환경에서 HTTP/1.1 강제 비교 테스트를 수행한다.
- 동기 `httpx` 호출을 `async def` 라우터에서 직접 실행한다면 이벤트 루프 차단 여부를 확인한다.

#### 예외 분류

| 예외 | 권장 응답 | 내부 코드 |
|---|---:|---|
| 원격 연결 종료 | 502 또는 503 | `UPSTREAM_DISCONNECTED` |
| 외부 응답 시간 초과 | 504 | `UPSTREAM_TIMEOUT` |
| 외부 인증 실패 | 502 | `UPSTREAM_AUTH_ERROR` |
| 내부 로직 오류 | 500 | `INTERNAL_ERROR` |

## 10. 캘린더 조회 최적화

### 점검 항목

- `user_id`, `scheduled_date`, `goal_id`, `is_completed` 인덱스
- 날짜 컬럼에 함수나 문자열 변환을 적용해 인덱스가 무효화되는지
- 목표별 마일스톤 N+1 조회 여부
- 전체 데이터를 가져온 뒤 Python에서 월 범위를 필터링하는지
- 같은 월 데이터를 사용자 동작마다 재조회하는지
- 동일 API가 여러 컴포넌트에서 중복 호출되는지

### 인덱스 후보

실제 스키마와 실행계획을 확인한 후 적용한다.

```sql
CREATE INDEX IF NOT EXISTS idx_milestones_user_scheduled_date
ON milestones (user_id, scheduled_date);

CREATE INDEX IF NOT EXISTS idx_milestones_goal_id
ON milestones (goal_id);

CREATE INDEX IF NOT EXISTS idx_goals_user_id
ON goals (user_id);
```

월 조회는 날짜 컬럼에 함수를 씌우기보다 범위 조건을 사용한다.

```sql
WHERE user_id = :user_id
  AND scheduled_date >= :month_start
  AND scheduled_date < :next_month_start
```

### 캐시와 데이터 재사용

- 같은 사용자·월 데이터는 클라이언트 캐시에 유지한다.
- 변경된 월 캐시만 무효화한다.
- 월 데이터에 상세 패널 정보가 있다면 날짜 선택 시 별도 일 조회를 생략한다.
- 추가 데이터가 필요해도 패널은 즉시 열고 상세 정보만 지연 로딩한다.

## 11. 프론트엔드 상태 갱신

### 낙관적 업데이트

마일스톤 체크는 클릭 즉시 로컬 UI를 변경하고 실패 시 롤백한다.

```text
클릭
→ 로컬 UI 즉시 체크
→ PATCH 요청
→ 성공: 서버와 동기화
→ 실패: 이전 상태 복원 + 오류 표시
```

현재 약 6초인 토글 체감시간을 수십 ms의 시각 반응으로 줄일 수 있다. 서버 확정 시간은 별도로 측정한다.

### 생성 결과 직접 반영

POST 응답으로 생성된 전체 객체를 반환하고 목록과 캘린더 캐시에 즉시 추가한다. 생성 후 `/goals`, `/milestones/today`, `/calendar/month`, `/calendar/date`, `/settings`를 모두 다시 호출하지 않는다.

### 중복·경쟁 제거

| 현재 | 개선 |
|---|---|
| 변경마다 전체 데이터 재조회 | 관련 리소스와 캐시만 갱신 |
| 컴포넌트마다 동일 API 호출 | 공유 쿼리 캐시 사용 |
| 이전 날짜 요청 계속 실행 | `AbortController`로 취소 |
| 응답 순서와 무관하게 반영 | 요청 ID 또는 선택 날짜가 최신일 때만 반영 |

## 15. 포트폴리오 활용 문장

### 문제 정의

> Playwright로 목표 생성, 마일스톤 생성·완료, 설정 변경, 날짜 선택 흐름을 각 25회 반복 측정한 결과 전체 성공률은 91.2%였고, API 823건 중 36건에서 502가 발생했습니다. 특히 월·일 캘린더 조회의 정상 응답 P95가 각각 약 11.35초와 10.42초로 나타나 사용자 인터랙션 지연의 핵심 병목으로 확인했습니다.

### 원인 분석

> FastAPI 로그를 추적해 Supabase 통신 과정의 HTTP/2 연결 종료(`httpcore.RemoteProtocolError: Server disconnected`)가 조회 API의 502로 전파되는 것을 확인했습니다. 또한 사용자 동작당 평균 6.58회의 API 호출이 발생하며 전체 데이터 재조회와 중복 캘린더 호출이 연결 부하 및 UI 갱신 지연을 확대하고 있음을 분석했습니다.

### 개선 성과 작성 틀

> 외부 연결 예외에 대한 제한적 재시도, 중복 조회 제거, 캘린더 쿼리 및 인덱스 최적화, 낙관적 UI 업데이트를 적용했습니다. 동일 조건의 Playwright 반복 테스트에서 `[기존 P95]ms → [개선 P95]ms`로 `[개선율]%` 단축했고, 전체 성공률을 `91.2% → [개선 후 성공률]%`, 502 비율을 `4.37% → [개선 후 비율]%`로 개선했습니다.

실제 재측정 전에는 목표 수치를 성과처럼 작성하지 않는다.

## 16. 결론

현재 문제는 다음 세 가지로 요약된다.

1. Supabase 통신 중 HTTP/2 원격 연결이 종료되어 FastAPI 조회 API가 502를 반환한다.
2. 월·일 캘린더 조회의 정상 응답 자체가 느리고 긴 꼬리 지연을 보인다.
3. 생성·토글 이후 광범위한 전체 재조회가 발생해 쓰기 API가 빠르더라도 최종 UI 반영은 수 초가 걸린다.

개선 순서는 **502 복구 처리 → 캘린더 쿼리 최적화 → 중복 조회 제거 → 낙관적 UI 업데이트 → 테스트 관측성 개선**이 적절하다. 특히 마일스톤 토글은 서버 완료를 기다리지 않고 즉시 시각적으로 반영하면 사용자 체감 성능을 가장 크게 개선할 수 있다.

이 문서는 개선 전 기준선으로 활용하며, 동일 조건 재측정 후 성공률, 502 비율, 시나리오별 P95, 캘린더 API P95를 비교해 포트폴리오에 정량적인 개선 성과를 제시한다.

| Bing 사례 | 추가 지연 **1.5초** | 검색 수 **1.8% 감소**, 매출 **4.3% 감소**, 만족도 **3.8% 감소** | 명시적인 불만이 나타나기 전에도 사용 행동은 악화될 수 있음 |
| Google 사례 | 추가 지연 **400ms** | 사용자당 검색 수 **0.59% 감소** | 수백 ms 수준의 지연도 반복 사용 행동에 영향을 줄 수 있음 |

#### 2. MileDay 적용 방향

MileDay의 목표 생성, 마일스톤 완료, 날짜 선택은 사용자가 즉각적인 반응을 기대하는 단순 상호작용이다. 따라서 논문에서 확인된 7~10초 수준의 사용자 한계를 허용 기준으로 사용하지 않고, 모든 핵심 동작이 **3초 이내에 완료되거나 오류를 반환하도록 제한**한다.

또한 3초 이내에 끝나는 작업에는 초 단위 경과시간이나 카운트다운을 표시하지 않는다. 대신 사용자의 입력을 즉시 화면에 반영하고, 처리가 1초를 넘을 때만 현재 상태를 간단히 안내한다.

| 경과시간 | MileDay 동작 기준 | UI 처리 |
|---:|---|---|
| **0~100ms** | 즉각 반응 구간 | 클릭, 체크, 날짜 선택 상태를 즉시 시각적으로 반영 |
| **100~500ms** | 목표 체감 응답시간 | 별도 로딩 표시 없이 정상 처리 |
| **500ms~1초** | 허용 지연 구간 | 기존 화면을 유지하며 요청 처리 |
| **1~2초** | 상태 안내 구간 | `저장 중`, `일정을 불러오는 중`과 같은 메시지와 스피너 표시 |
| **2~3초** | 지연 경고 구간 | 중복 클릭 방지, 요청 완료 또는 실패 여부 확인 |
| **3초 초과** | 실패 처리 구간 | 요청 종료, 오류 메시지와 재시도 기능 제공 |
| **7초 이상** | 허용 불가 | 사용자에게 노출되지 않도록 클라이언트·서버 타임아웃 적용 |

#### 5. 적용 결론

MileDay는 실제 처리시간과 지각된 대기시간을 함께 관리한다. 사용자의 입력은 100ms 이내에 시각적으로 반영하고, 1초 이상 지연될 때만 상태 메시지를 표시한다. 모든 핵심 요청은 3초 이내에 완료하거나 오류를 반환하도록 제한하며, 3초를 초과하면 무기한 로딩 대신 재시도 가능한 오류 메시지를 제공한다.

특히 마일스톤 완료와 같은 단순 상호작용에는 낙관적 업데이트를 적용하여 네트워크 처리시간과 관계없이 즉각적인 반응을 제공한다. 이를 통해 사용자가 시스템의 정상 작동 여부를 추측하며 기다리는 상황을 줄이고, 응답 지연이 앱의 만족도와 반복 사용 행동에 미치는 영향을 최소화한다.
