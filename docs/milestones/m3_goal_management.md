# M3. 목표 관리 및 반복 설정 로직

## 문서 목적

M3 단계에서는 M2에서 구현한 인증 기반 위에 목표 관리 백엔드 흐름을 실제 동작하도록 구현한다.
목표 API는 더 이상 임시 응답을 반환하지 않고, 인증된 현재 사용자 ID를 기준으로 Supabase `goals` 테이블에 목표를 저장하고 조회한다.
현재 DB 스키마가 제공하는 `is_recurring`, `recurrence_type` 범위 안에서 반복 설정을 검증한다.

## 구현 범위

| 요구사항 | 구현 내용 |
| --- | --- |
| BE-03 | 실제 `/goals` CRUD 라우터를 service/repository 계층에 연결 |
| GL-01 | 제목, 마감일, 색상, 반복 설정을 포함한 목표 생성 |
| GL-02 | 현재 사용자의 목표 목록 조회 |
| GL-03 | 목표 필드 수정 |
| GL-04 | 현재 사용자의 목표 삭제. 하위 마일스톤 삭제는 DB FK cascade 기준 유지 |
| GL-05 | `goal_id` 기준 현재 사용자 목표 상세 조회 |
| GL-06 | 반복 목표 설정 검증 |
| RP-01 | API 스키마에서 `daily`, `weekly`, `monthly` 반복 유형 지원 |
| RP-02 | 반복 날짜별 마일스톤 생성 방식으로 처리 예정. 별도 반복 예외 테이블은 만들지 않음 |

## 구현 내용

- `backend/app/api/routers/goals.py`의 모든 목표 API에 `require_current_user_id` 인증 의존성을 적용했다.
- `backend/app/services/goal_service.py`에서 목표 소유자 확인, 반복 설정 검증, 예외 변환을 담당한다.
- `backend/app/repositories/goals.py`에서 Supabase `goals` 테이블 접근을 감싸고, 조회/수정/삭제에는 항상 `user_id` 조건을 적용한다.
- `backend/app/schemas/goal_schemas.py`에서 잘못된 반복 설정 조합을 거부한다.
  - 반복 목표는 `recurrence_type`이 반드시 필요하다.
  - 반복하지 않는 목표는 `recurrence_type`이 `null`이어야 한다.
  - 반복 유형은 `daily`, `weekly`, `monthly`만 허용한다.
- `backend/app/exceptions/handlers.py`에서 Pydantic 검증 상세 정보에 예외 객체가 포함되어도 JSON 응답 직렬화가 실패하지 않도록 처리했다.

## 반복 목표 처리 기준

반복 예외 날짜는 별도 테이블에 저장하지 않는다.
MVP에서는 반복 목표를 생성할 때 반복 기간 안의 각 회차를 실제 `milestones` row로 미리 생성하는 방식을 사용한다.

기준 흐름은 다음과 같다.

1. 사용자가 반복 목표를 생성한다.
2. 백엔드가 반복 시작일, 마감일, `recurrence_type`을 기준으로 반복 날짜 목록을 계산한다.
3. 계산된 각 날짜마다 `milestones` row를 생성한다.
4. 사용자가 특정 날짜를 예외로 지정하면, 해당 날짜의 자동 생성 마일스톤을 생성하지 않거나 이미 생성된 마일스톤을 삭제한다.
5. 캘린더, 날짜 상세, Today List는 별도 반복 계산 없이 `milestones.scheduled_date`만 조회한다.

이 방식의 장점은 다음과 같다.

- 캘린더 조회와 Today List 조회가 단순해진다.
- 반복 예외 처리는 해당 날짜의 마일스톤 삭제로 처리할 수 있다.
- 반복 회차별 완료 처리와 개별 수정이 자연스럽다.
- 현재 DB 스키마를 크게 변경하지 않아도 된다.

반복 목표 수정 기준은 다음처럼 둔다.

- 반복 설정 변경 시 과거 마일스톤과 완료된 마일스톤은 보존한다.
- 아직 완료되지 않은 미래 마일스톤만 재생성 대상으로 삼는다.
- 사용자가 직접 수정한 마일스톤을 자동 생성 마일스톤과 구분하려면 후속 스키마 보강이 필요하다.

현재 스키마에는 반복 시작일과 예외 날짜 입력 필드가 없다.
따라서 실제 구현 전에 다음 입력 기준을 먼저 확정해야 한다.

- 반복 시작일을 목표 생성일로 볼지, 별도 `start_date` 필드를 둘지
- 예외 날짜를 목표 생성 요청에 포함할지, 별도 API로 받을지
- 자동 생성 마일스톤 제목을 목표 제목과 동일하게 둘지, 별도 템플릿을 둘지

## 검증 내용

추가한 테스트는 다음 항목을 검증한다.

- 인증 없이 `/goals` 접근 시 `401` 반환
- 라우터 계층이 현재 사용자 ID를 목표 서비스에 전달하는지 확인
- 목표 생성, 목록 조회, 상세 조회, 수정, 삭제 라우트 동작
- 반복 목표 생성과 잘못된 반복 설정 요청 검증
- 서비스 계층의 사용자 소유권 확인
- 현재 저장 상태를 기준으로 한 부분 반복 설정 수정 검증
- 목표 생성/수정/삭제 실패 예외 변환
- repository 계층의 Supabase query에 `id`, `user_id` 필터가 적용되는지 확인
- Supabase 응답이 비어 있을 때 상세 조회/생성/수정/삭제 결과 처리

## 후속 처리 항목

반복 마일스톤 자동 생성은 다음 단계에서 구현한다.
이때 별도 반복 예외 테이블을 만들기보다, 반복 날짜별 마일스톤을 materialize하고 예외 날짜의 마일스톤을 생성하지 않거나 삭제하는 방식으로 진행한다.

구현 시 추가로 검증할 항목은 다음과 같다.

- `daily`, `weekly`, `monthly` 반복 날짜 계산
- 반복 목표 생성 시 마감일 안의 마일스톤 일괄 생성
- 예외 날짜가 지정된 경우 해당 날짜 마일스톤 미생성
- 이미 생성된 반복 마일스톤의 예외 처리 시 해당 마일스톤 삭제
- 반복 설정 수정 시 미완료 미래 마일스톤만 재생성
