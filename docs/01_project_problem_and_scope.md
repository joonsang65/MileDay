## 1. 문제 정의

현재 사용 중인 Windows 위젯 앱의 사용성이 떨어진다.

### 기존 위젯/캘린더 앱 사용 중 불편한 점

- 매주 반복하는 작업 중 특정 주에는 하지 않아도 되는 일정을 지우고 싶은데, 그 일정을 지우면 전체 반복 일정이 삭제된다.
- 글꼴이나 UI를 바꾸려면 유료 버전을 구매해야 한다.
- 공휴일도 주말처럼 처리하고 싶은데, 해당 기능을 지원하지 않는다.
- 설정 번역이 제대로 되어 있지 않아 원하는 기능이 있는지 알기 어렵다.
- 해야 하는 작업별로 색상을 지정하고 싶은데, 일일이 지정해야 해서 번거롭다.

### 추가로 필요하다고 판단한 기능

- 글자로 하나하나 쳐서 정리하기보다, 자연어로 일정 설정 요구사항을 전달하면 일정을 자동으로 변경해주는 방식이 있으면 좋다.
- 해야 할 일을 수행하려면, 단계를 나누어 어디까지 진행했는지 추적할 수 있으면 좋다.
- UI를 가능한 사용자가 원하는 대로 수정할 수 있으면 좋다. 단, 직관적인 방식이어야 한다.

## 2. 기존 서비스 한계

| 서비스 | 주요 용도 | 장점 | 사용자들이 경험하는 한계 | MileDay에 반영할 시사점 |
|---|---|---|---|---|
| Outlook Calendar | Microsoft 기반 일정 관리, Windows 일정 확인 | Microsoft 365 및 Outlook 계정과의 연동이 자연스럽다. 별도 설치 없이 접근하기 쉽다. 작업 표시줄 날짜 클릭이나 앱을 통해 일정을 빠르게 확인 가능하다. | 바탕화면에 상시 노출되는 위젯 형태가 아니다. 디자인이나 배치 커스터마이징이 어렵다. 세부 일정 관리 중심이므로 목표와 하위 작업을 구조화해 관리하기 어렵다. | 단순 일정 확인이 아니라 바탕화면에 계속 보이는 위젯형 플래너로 구현할 필요가 있다. 목표 → 마일스톤 → 오늘 할 일 구조를 명확히 보여주는 방식으로 차별화해야 한다. |
| One Calendar | 여러 캘린더 계정 통합 관리 | 여러 캘린더를 한 곳에서 통합해 볼 수 있다. 다양한 외부 캘린더 계정 연동에 강점이 있다. | 여러 캘린더를 통합해 보여주는 데 초점이 맞춰져 있다. 개인 목표를 세부 작업으로 나누고 진행 상태를 관리하는 프로젝트형 사용에는 한계가 있다. 동기화와 다중 기기 연동은 편리하지만 목표 기반 작업 관리는 제한적이다. | MVP 단계에서는 외부 캘린더 통합보다 자체적인 목표 및 마일스톤 관리 구조에 집중한다. 먼저 Goal/Milestone 데이터 모델을 안정적으로 설계하고 외부 캘린더 연동은 추후 기능으로 분리한다. |
| Me Calendar | Windows 네이티브 캘린더, 데스크톱 위젯 | Google, Outlook, Microsoft 365, iCloud 등 여러 캘린더 계정을 지원한다. 위젯을 통해 앱을 직접 열지 않아도 데스크톱에서 일정을 확인하고 관리할 수 있다. | 투명도 기능 오류로 일부 사용자는 위젯을 정상적으로 사용하지 못하는 경우가 있다. 일부 장치에서는 투명도 효과가 적용되지 않는 문제가 언급된다. 위젯 위치 관리, 듀얼 모니터 환경, 팝업 처리 등 데스크톱 위젯 구현 자체의 복잡도가 높다. | 초기 MVP에서는 투명도, 듀얼 모니터 대응, 고급 위젯 효과를 과하게 구현하지 않는다. 창 위치 저장, 크기 조정, 항상 위 토글 등 핵심 위젯 동작을 우선 구현하고, 복잡한 위젯 효과는 별도 이슈로 관리한다. |

## 3. 요구사항 헤드라인 정리

1. 로그인 시스템으로 사용자를 관리한다.
2. 사용자가 직접 목표를 설정하고 관리하도록 한다.
3. 목표 안에는 세부 마일스톤을 두고, 이를 실제 TODO로 설정하도록 한다.
4. TODO는 캘린더에서 확인할 수 있고, TODO 외에도 공휴일이나 설정한 내용을 확인할 수 있다.
5. 날짜를 선택하면 해당 날짜에 사용자가 설정한 내용을 확인할 수 있다.
6. 캘린더 아래에 오늘 할 일 리스트를 확인할 수 있다.
7. 위젯 형태로 보여지고, 창 레이아웃은 사용자가 변경 가능해야 한다.
8. 설정 부분은 직관적으로 구성한다.
9. 반복 설정이 가능하고, 반복 중 특정 날짜만 제거할 수 있어야 한다.
10. FastAPI 기반 백엔드를 구현하고, FE 요청을 BE가 관리한다.
11. DB는 Supabase를 사용하고, 목표, 마일스톤, 사용자 테이블로 구분한다.
12. RLS를 적용해서 사용자 간 목표가 섞이지 않도록 한다.
13. 동작 과정에서 에러가 발생하는 부분을 로그로 확인 가능하게 설정한다.
14. 위 내용이 정리되면 외부 캘린더에 저장된 내용도 동기화 가능하도록 설정한다.
15. 배포는 Windows 실행 파일로 패키징하고, 나중에 해당 앱과 연동되는 모바일 앱도 구현한다.
